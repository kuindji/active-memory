An entry is superseded when the new entry corrects, updates, reverses, or replaces the existing one — for example, a new decision that reverses an earlier one, a new rationale that invalidates the old reasoning, or a new direction that overrides a previous plan.

Only flag true supersession — not mere similarity, overlap, or elaboration. If the new entry merely adds detail or provides an example, that is NOT supersession (it's a related entry, which is handled separately).

Common supersession patterns in code repos:
- "We decided to use X" → later "We are switching from X to Y"
- "The auth module lives in src/auth" → later "We moved the auth module to src/services/auth"
- "Use Postgres for the orders table" → later "Migrated orders to DynamoDB for write throughput"
